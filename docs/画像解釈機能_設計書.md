# 画像解釈機能 設計書（素案 v0.2）

## 1. 文書目的
要件定義書に基づき、MVP実装に必要なアーキテクチャ、モジュール責務、インターフェース、例外/ログ/設定を具体化する。

## 2. 設計方針
- 既存Streamlitアプリに新規ページを追加して機能提供。
- **UI / Application Service / Model Client** の3層分離。
- モデル依存部分をアダプタ化し、将来のプロバイダ変更を容易化。
- 入力値検証をサービス層に集約し、UIは表示責務に限定。

## 3. 想定ディレクトリ構成

```text
app.py
pages/
  pageImage.py
src/
  image_interpretation/
    __init__.py
    schemas.py
    errors.py
    service.py
    model_client.py
    logging_utils.py
```

## 4. コンポーネント責務

| ファイル | 役割 | 主な責務 |
|---|---|---|
| `pages/pageImage.py` | UI層 | 入力受付、実行トリガ、結果/エラー表示 |
| `service.py` | アプリ層 | バリデーション、プロンプト組み立て、モデル呼び出し制御 |
| `model_client.py` | インフラ層 | VisionモデルAPIとのI/O変換、タイムアウト設定 |
| `schemas.py` | 共有スキーマ | Request/Responseデータ構造 |
| `errors.py` | 例外定義 | ドメイン例外の定義と分類 |
| `logging_utils.py` | 横断関心 | request_id採番、構造化ログ補助 |

## 5. 処理シーケンス
1. UIが画像ファイルとプロンプトを受け取る。
2. UIが `InterpretationRequest` を組み立て `service.interpret_image()` を呼び出す。
3. Serviceが入力検証（拡張子/サイズ/空入力）を行う。
4. Serviceが `request_id` を生成し、開始ログを記録。
5. Serviceが `VisionModelClient.generate_description()` を実行。
6. Serviceが結果を `InterpretationResult` に変換。
7. Serviceが終了ログ（成功/失敗）を記録。
8. UIが結果またはエラー文言を表示。

## 6. データ設計

### 6.1 Request
```python
InterpretationRequest(
    image_bytes: bytes,
    filename: str,
    mime_type: str,
    instruction_prompt: str | None,
)
```

### 6.2 Response
```python
InterpretationResult(
    description_text: str,
    model_name: str,
    latency_ms: int,
    request_id: str,
)
```

## 7. インターフェース設計

### 7.1 `service.py`
- `validate_image(filename: str, size_bytes: int) -> None`
- `build_prompt(user_prompt: str | None) -> str`
- `interpret_image(req: InterpretationRequest) -> InterpretationResult`

### 7.2 `model_client.py`
- `class VisionModelClient:`
  - `generate_description(image_bytes: bytes, prompt: str) -> str`

## 8. バリデーション設計
- 許可拡張子: `.png`, `.jpg`, `.jpeg`, `.webp`
- 最大サイズ: `MAX_IMAGE_SIZE_MB`（初期値10MB）
- ファイル0バイト拒否
- プロンプト最大長: 2,000文字
- 空プロンプトはデフォルト値へフォールバック

## 9. 例外設計

| 例外名 | 発生条件 | UI表示方針 |
|---|---|---|
| `InvalidImageFormatError` | 未対応拡張子 | 対応形式を明示して案内 |
| `ImageTooLargeError` | サイズ上限超過 | 上限値を表示 |
| `InvalidPromptError` | 文字数超過等 | 入力制限を案内 |
| `ModelTimeoutError` | APIタイムアウト | 再試行を促す |
| `ModelUnavailableError` | API障害 | 一時的エラーとして通知 |

## 10. ログ設計（構造化）
- 必須キー: `request_id`, `event`, `status`, `latency_ms`, `error_type`, `model_name`
- イベント例:
  - `image_interpretation.start`
  - `image_interpretation.success`
  - `image_interpretation.failure`

## 11. 設定値設計

| 環境変数 | 必須 | 初期値 | 用途 |
|---|---|---|---|
| `VISION_API_KEY` | Yes | - | モデルAPIキー |
| `VISION_MODEL_NAME` | Yes | - | 使用モデル名 |
| `VISION_TIMEOUT_SEC` | No | 10 | API呼び出しタイムアウト |
| `MAX_IMAGE_SIZE_MB` | No | 10 | 画像サイズ上限 |

## 12. セキュリティ設計
- APIキーは環境変数のみから読み込み。
- ログに画像バイナリ・個人情報を含めない。
- ユーザー画面に内部例外詳細（スタックトレース）を表示しない。
- 必要に応じて将来、画像のPII検知・マスキングを追加。

## 13. テスト設計（MVP）

### 13.1 単体テスト
- `validate_image` 正常/異常（形式、サイズ、空ファイル）
- `build_prompt` のフォールバック確認
- 例外マッピング（ModelTimeoutErrorなど）

### 13.2 結合テスト
- UI入力→Service→ModelClient（モック）→結果表示
- API失敗時にUIへ適切なエラーメッセージ表示

### 13.3 受け入れテスト
- 要件定義書の AC-01〜AC-05 を満たすこと
